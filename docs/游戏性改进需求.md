# 游戏性改进需求文档

> 日期: 2026-06-17
> 状态: 制作中
> 目标: 把项目从"技术扎实但留不住人的 demo"打磨成"有爽感、有节奏、有反馈的可玩游戏"

---

## 背景与问题诊断

当前项目架构完整(9 引擎 / 133 测试 / 事件总线 / 三层情绪 / 关联矩阵),但存在四个影响留存的核心问题:

1. **操盘名不副实** — 玩家交易对价格冲击极弱(`netFlow = (buy-sell)/liquidity * 0.001`,投 100 万才动 ~0.01%)。价格主要由 AI/事件驱动,玩家是"下注者"而非"操盘手"。
2. **没有目标节奏** — sandbox 式开局,玩家面对一堆面板却不知道"现在该干什么",缺乏阶段感和成就感。
3. **反馈缺失** — 关键动作(交易获利、操纵累积热度、造假被揭穿)缺少即时视觉/文字反馈,玩家感知不到因果。
4. **学习门槛高** — 8 个 tab + 6 角色 + 3 国家子系统,对新人信息过载。

---

## 改进方案(按优先级)

### P1 — 放大玩家交易冲击(核心爽感)

**问题**: 玩家交易不影响价格,违背"金融操盘"的核心预期。

**方案**:
- 提升 `MarketEngine` 的 `netFlow` 系数:`0.001` → `0.01`(10 倍),让中等仓位也能推动 ~0.5-1% 的单 tick 波动
- 新增**主动操纵技能「大单砸盘 / 拉升」**: 玩家可一次性注入大额定向资金流,产生明显的即时价格冲击(消耗操纵热度,有冷却)。挂在右侧交易面板。
- 调整 AI 交易量基线,避免 AI 完全主导价格(适度下调 AI 单笔交易额)

**预期效果**: 玩家能"砸盘"赚到做空的快感,操盘行为名副其实。

**改动文件**: `MarketEngine.ts`(系数)、`RightPanel.tsx`(新按钮)、`useGameLoop.ts`(wrapper)、`playerStore` 或新引擎(冷却)

---

### P2 — 目标与节奏系统

**问题**: 没有明确目标,玩家不知道该干什么。

**方案**: 引入**动态目标系统**,在顶栏或独立面板展示,完成即奖励 + 推进阶段:
- **阶段化资金目标**:每个角色有 3-4 档里程碑(如起始资金 100 万 → 300 万 → 1000 万 → 5000 万),达成解锁段位/成就/音效
- **任务池**:随机生成限时小任务,如"30 秒内做空 OIL 获利 5 万"、"发布 1 条未被揭穿的假新闻"、"雇佣任意 KOL"
- **目标进度条**:顶栏显示当前阶段进度,给游戏一个"spine"

**预期效果**: 玩家始终有"下一步",从漫无目的变成有方向推进。

**改动文件**: 新建 `engine/objective/ObjectiveEngine.ts` + `stores/objectiveStore.ts` + 顶栏/面板 UI

---

### P3 — 即时反馈层(Toast 通知)

**问题**: 关键动作缺少即时回响,玩家感知不到因果。

**方案**: 扩展现有 `Toast` 系统,对以下事件弹非阻塞通知:
- 交易平仓:显示实现盈亏(`+$50,000 做空 USD 获利`)
- 操纵热度变化超阈值:`⚠ 操纵热度 18,接近危险`
- 造假被揭穿:`🎭 关于 AAPL 的假新闻被识破,价格回调`
- 拍卖/情报兑现:`🛢 沙特拍卖揭晓: OIL ▲ 5.2%`(玩家若持有/竞标了内幕)
- 目标达成:`🏆 资产突破 300 万!`

**预期效果**: 每个动作都有回响,正反馈循环建立。

**改动文件**: `useGameLoop`(监听事件触发 toast)、`Toast.tsx`(增强)、新事件监听

---

### P4 — 新手引导(渐进解锁)

**问题**: 8 个 tab + 多系统同时开放,信息过载。

**方案**: 基于 `unlockStore` 做渐进解锁(保留现有解锁框架):
- **第一局**:只开放 交易 + 造假 + 新闻/交易/舆论 tab
- **第二局**(或资产达 200 万):解锁 KOL + 发币
- **第三局**(或破产 1 次):解锁 拍卖 + 情报 + 离岸
- 首次进入游戏自动弹出简化版 HelpOverlay,只讲"交易+造假"

**预期效果**: 降低首屏认知负荷,让玩家循序渐进掌握系统。

**改动文件**: `unlockStore.ts`(扩展解锁条件)、`LeftPanel.tsx`(按解锁状态渲染 tab)、`App.tsx`(首次引导)

## 不做的事(明确边界)
- 不重写引擎架构(P1-P4 都在现有架构上增量)
- 不引入新依赖(framer-motion 等保持不用)
- 不改测试断言的语义(只同步受影响的 id)

## 验证方式
- 每完成一个 P 跑一次 `npm test`,确保 133 测试不回归
- P1/P2 完成后人工试玩验证爽感/节奏
- 最终全部完成后提交推送

---

## 制作顺序
**P1 → P2 → P3 → P4**(按"提升留存的性价比"排序)。P1 是最根本的(让操盘有爽感),优先做。
